# 📘 부기맨(Boogieman) 도서 관리 시스템 요구사항 정의서

## 1. 프로젝트 개요
**부기맨(Boogieman)**은 사내 구성원이 원하는 희망 도서를 편리하게 신청하고, 회사가 보유한 도서를 쉽게 검색하여 대출/반납할 수 있도록 지원하는 사내 도서 복지 관리 시스템입니다.

**최소 기능 제품(MVP) 목표:** 사원들이 도서를 신청하고, 실물 도서를 대출/반납하며, 보유 도서를 검색하는 "핵심 도서 순환 사이클"이 우선 오동작 없이 운영되는 것을 MVP 목표로 합니다. (R1 기능 중심)

## 2. 목표 사용자
* **일반 사원 (User)**: 희망 도서 신청, 도서 검색 및 대출/반납 기능, 마이페이지 대출/예약 현황 조회 기능 사용
* **관리자 (Admin)**: 신청 도서 승인/거절, 신규 도서 입고(등록), 도서 상태 현황 관리, 연체 및 예약 취소/승계 관리

## 3. 기술 스택

| Concern | Technology |
|---------|-----------|
| Database | MariaDB 11.3 |
| Cache | Redis 7 |
| Migrations | Flyway (`src/main/resources/db/migration/`) |
| Auth | JWT (JJWT 0.11.5) + Kakao OAuth |
| Complex queries | QueryDSL |
| API docs | SpringDoc OpenAPI (Swagger UI at `/swagger-ui.html`) |
| Templates | Thymeleaf |

## 4. 제약 조건 (보안/성능/비즈니스 룰)

### 🔒 보안 (Security & Safety)
1. **IDOR (접근 제어 취약점) 방지 철저**: 예약 취소 통신이나 도서 반납 기능의 엔드포인트에서 타인의 `borrowId`, `reservationId`를 파라미터로 무단 지정하여 호출하는 악의적 조작이 불가능하도록, 백엔드에서 인증 토큰(세션 권한) 소유 여부를 기반하는 검증 과정 (403 Forbidden 등으로 블록)이 필수적입니다.
2. **CSRF, XSS 방어**: 도서 신청 사유 등 사용자의 텍스트 입력 구간에 악성 스크립트 작성 시도를 막기 위한 XSS 필터링 적용.
3. **Hard Delete 사용 금지 (Soft Delete 전환 제안)**: 회원이 부서를 이동/탈퇴하거나 도서가 분실/폐기되었을 때 테이블 레코드를 삭제(`DELETE`)하는 것이 아닌 `is_deleted` 컬럼이나 `status="REMOVED"`로 설계하여 시스템 안의 과거 "예전 통계 및 이력"이 깨지는 데이터 무결성 훼손을 방지합니다.

### ⚡ 성능 (Performance) 및 예외 제어, 동시성
1. **도서 대출/예약 락 (Lock) 처리 (동시성 방어)**: 이벤트 기간이나 신간 입고 시 잔여 수량이 1권인 인기 서적에 다수의 트랜잭션이 수 밀리초 차이로 발생한다면 DB가 여러 명에게 모두 빌려준 것으로 오인할 수 있습니다. **Pessimistic DB Lock (비관적 락)** 이나 **Optimistic Lock (@Version)** 방식의 로직 방어를 통해 1인에게만 도서를 안전하게 할당하고 나머지는 대출 실패 처리를 내려합니다.
2. **비동기 메시지/알림 응답 속도 보완**: 도서 예약의 자동 통보 및 반납 독촉 진행 시 카카오톡이나 이메일 등의 외부 API 요청으로 서비스 전체 로딩이 병목 현상에 빠져 UX가 손상되지 않도록 대응해야 합니다. 발송 로직 자체는 메인 스레드를 차지하지 않는 **Spring ApplicationEventPublisher** 또는 **스레드 풀(@Async)**을 활용하여 핵심 로직에서 분리하는 구조가 권장됩니다.
